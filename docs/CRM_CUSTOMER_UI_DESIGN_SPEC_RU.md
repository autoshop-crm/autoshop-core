# CRM Customer UI Design Spec

Дата: 2026-05-14  
Статус: design-direction approved by deep interview  
Проект: `FrontClient`

---

## 1. Цель документа

Этот документ описывает **дизайн клиентской CRM с точки зрения UI**.

Он нужен, чтобы:

- держать единое визуальное направление проекта;
- не скатиться в копию staff CRM;
- зафиксировать screen patterns, visual rules и information hierarchy;
- дать основу одновременно для web-версии и будущего mobile-приложения.

---

## 2. Главный UI-вектор

Основной эффект интерфейса:

- **скорость и удобство**

Это означает, что UI должен в первую очередь помогать клиенту:

- быстро понять, что происходит;
- быстро найти нужную информацию;
- быстро сделать ключевое действие;
- не тратить внимание на лишнее.

### Что это значит практически

Интерфейс должен ощущаться как:

- app-like;
- карточный;
- мобильный по логике;
- без лишней декоративности;
- с общим визуальным языком для web и mobile.

Это не “desktop CRM, адаптированная под телефон”. Это единая продуктовая UI-система, которая должна одинаково естественно масштабироваться в браузере и в мобильном приложении.

---

## 3. Основные UI-принципы

## 3.1. Action-first

Каждый экран должен помогать сделать следующее:

- понять статус;
- понять цену;
- понять следующее действие;
- открыть нужную сущность;
- подтвердить решение.

## 3.2. Не больше нужного

На экране не должно быть “всего сразу”.

Правило:

- не более 7 важных объектов внимания одновременно;
- если блок не помогает принять решение или понять состояние, он должен уходить ниже или исчезать.

## 3.3. Не более 3 кликов до нужной информации

Основная информация должна быть достижима максимум за 3 действия:

- dashboard -> order details
- orders list -> конкретный заказ
- dashboard -> approvals -> approve/reject
- vehicles -> vehicle details -> order history

## 3.4. Общий язык для web и mobile

Web UI и mobile app должны использовать одну и ту же основу:

- одинаковую иерархию;
- одинаковые статусы;
- одинаковые паттерны карточек;
- одинаковую логику CTA;
- одинаковую плотность и ритм.

Различаться могут только layout mechanics, но не продуктовая логика.

## 3.5. No operational noise

Клиент видит только то, что влияет на него.

Не показывать в первом слое:

- внутренние operational детали;
- служебные идентификаторы;
- внутренние роли сотрудников;
- закупочную и складскую механику;
- технические enum-статусы;
- внутреннюю CRM-терминологию.

---

## 4. Желаемый характер интерфейса

UI должен быть:

- быстрым;
- лёгким;
- современным;
- спокойным;
- предсказуемым;
- без визуального шума.

### Не должен быть

- тяжёлым enterprise-dashboard;
- “офисной CRM” для клиента;
- переусложнённым;
- декоративным ради декоративности;
- dense table-first интерфейсом.

---

## 5. Visual System Direction

## 5.1. Общая стилистика

Рекомендуемая стилистика:

- светлый фон;
- чистые карточки;
- мягкие радиусы;
- заметные, но не агрессивные CTA;
- простая и сильная типографика;
- ограниченная палитра;
- акцент на status blocks, progress blocks и action cards.

## 5.2. Визуальная плотность

Плотность должна быть ниже, чем в staff CRM.

Нужно:

- больше воздуха между блоками;
- более крупные tappable areas;
- меньше одновременно видимых второстепенных деталей;
- меньше табличности;
- больше вертикального flow.

## 5.3. Карточность как основной паттерн

Основной паттерн UI:

- карточки состояния;
- карточки действий;
- карточки сущностей;
- карточки summary;
- карточки timeline/events.

Таблицы допустимы только там, где без них реально хуже, но для MVP клиентского контура они не должны быть default-паттерном.

## 5.4. Типографика

Иерархия текста должна быть очень ясной:

- крупный заголовок состояния;
- короткий статус;
- короткое пояснение;
- заметный CTA;
- вторичный supporting text.

Текст должен быть коротким и утилитарным.

---

## 6. Information Architecture

## 6.1. Верхний уровень

Desktop navigation:

- Главная
- Мои заказы
- Мои автомобили
- Согласования
- Документы
- Лояльность
- Профиль

Mobile navigation:

- Главная
- Заказы
- Авто
- Согласования
- Профиль

## 6.2. Логика структуры

Приоритет разделов определяется не частотой данных, а частотой клиентского намерения:

1. Что сейчас с машиной?
2. Нужно ли что-то подтвердить?
3. Сколько это стоит?
4. Где мои машины и их история?
5. Где документы и бонусы?

---

## 7. Основные экранные паттерны

## 7.1. Dashboard

Главная страница должна быть не обзором разделов, а ответом на вопрос:

- что важно сейчас?

Приоритет блоков:

1. Активный заказ
2. Pending approval
3. Ближайшая запись
4. Быстрые действия
5. Автомобили / бонусы / документы как вторичный слой

### Что должен видеть пользователь сразу

- текущий статус;
- следующее действие;
- цену или её изменение, если важно;
- CTA, если нужен ответ клиента.

## 7.2. Orders List

Паттерн:

- список карточек, а не тяжёлая таблица;
- активные сверху;
- архив ниже;
- отдельный визуальный приоритет для заказов, где нужен action.

Карточка заказа должна показывать только essentials:

- номер / название;
- автомобиль;
- статус;
- краткий next step;
- цену / диапазон цены, если нужно;
- action marker.

## 7.3. Order Journey Screen

Это главный экран продукта.

Он должен давать за один экран:

- что происходит;
- что делать;
- сколько стоит;
- какие есть документы;
- нужно ли согласование;
- что было последним важным событием.

### Рекомендуемые блоки

1. Hero card
2. Status/progress card
3. Price summary card
4. Approval action card
5. Timeline preview card
6. Documents/photos preview card
7. Loyalty impact card

### Порядок блоков

Если есть pending approval, он поднимается выше timeline и документов.

## 7.4. Approvals UI

Approval screen — это action screen, а не информационная свалка.

Нужно показать:

- что найдено;
- зачем рекомендуем;
- сколько добавляется к цене;
- влияет ли это на срок;
- что произойдёт после решения;
- approve / reject.

### Mobile правило

На мобильном должна быть sticky action zone:

- `Подтвердить`
- `Отклонить`

## 7.5. Vehicles UI

Список машин должен быть лёгким и быстрым.

Карточка автомобиля:

- бренд / модель;
- номер / VIN;
- активные заказы, если есть;
- история заказов;
- CTA на запись или открытие истории.

## 7.6. Loyalty UI

Лояльность не должна выглядеть как бухгалтерия.

Показывать:

- баланс бонусов;
- текущую выгоду;
- связь бонусов с заказом;
- короткое объяснение пользы.

## 7.7. Documents UI

Документы и фото — это trust layer.

Нужны:

- понятные группы;
- превью;
- понятные CTA `Открыть` / `Скачать`;
- без сложной файловой терминологии.

---

## 8. Design Rules for Layout

## 8.1. Desktop

Desktop должен ощущаться как увеличенная app-surface, а не как сложная CRM-панель.

Нужно:

- умеренная ширина контента;
- сильная вертикальная структура;
- лёгкая навигация;
- карточная сетка;
- без перегруза боковыми utility-блоками.

## 8.2. Mobile

Mobile — первичный режим проектирования.

Нужно:

- bottom navigation;
- sticky actions на важных экранах;
- приоритет вертикального скролла;
- крупные tap targets;
- одна primary action на экранный момент.

---

## 9. Content Rules

Тексты в UI должны быть:

- короткими;
- понятными;
- без backend языка;
- ориентированными на действие.

### Вместо технических формулировок

Использовать:

- `Нужно ваше подтверждение`
- `Машина в работе`
- `Можно забрать автомобиль`
- `Есть обновление по заказу`
- `Стоимость изменилась`

Не использовать:

- raw enum names;
- служебные названия процессов;
- внутренние CRM labels.

---

## 10. Что UI намеренно скрывает

В клиентском UI первой версии скрываем:

- внутренние operational детали;
- внутренние workflow mechanics;
- детали staff-координации;
- служебные идентификаторы;
- данные, не влияющие на решение клиента.

Показываем только то, что реально важно клиенту:

- цена заявки;
- бонусы;
- его автомобили;
- его заказы;
- история заказов;
- согласования;
- документы;
- статус и следующее действие.

---

## 11. KPI качества UI-концепции

UI design spec считается удачным, если:

1. клиент делает ключевые действия быстро;
2. нужная информация достигается максимум за 3 клика;
3. на экране нет визуального мусора;
4. одновременно перед глазами не больше 7 важных объектов;
5. клиентский UI ощущается общим с будущим mobile app;
6. экран заказа не превращается в staff-style detail dump.

---

## 12. Прямые дизайн-следствия для реализации

На практике это означает:

- сначала проектировать mobile flow;
- затем масштабировать его в desktop;
- использовать cards как primary layout primitive;
- использовать CTA-first hierarchy;
- проектировать order screen как центр всей клиентской системы;
- все экраны проверять по правилам:
  - `не более 3 кликов`
  - `не более 7 важных объектов`
  - `нет operational noise`

---

## 13. Итоговое направление

Если описать UI проекта одной фразой:

> Это быстрый, app-like, карточный клиентский кабинет без лишней декоративности и без внутренней CRM-кухни, одинаково естественный для web и будущего mobile-приложения.
